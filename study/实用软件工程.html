# 实用软件工程

# 软件工程概述

# 软件的概念及特点

# 概念

  • 软件不仅包括程序，还包括程序处理对象——数据，以及程序开发、维护和使用有关的图文资料，即文档。

# 特点

  • 软件是一种逻辑实体具有抽象性。
  • 软件和硬件不同，是人类智慧劳动的产物。
  • 软件在运行过程中不会磨损。
  • 软件至今未摆脱手工艺开发的方式。
  • 软件的开发和运行必须依赖特定的操作系统。

# 软件的分类

# 基于功能的不同

  • 系统软件
  • 支撑软件
  • 应用软件

# 根据软件服务对象不同

  • 通用软件
  • 定制软件

# 按照软件产品规模不同

  • 小型软件
  • 中型软件
  • 大型软件
  • 超大型软件

# 根据工作的方式不同

  • 实时软件
  • 分时软件
  • 交互式软件
  • 批处理软件

# 软件危机的表现与原因

# 软件危机(课后题)

  • 指人们在开发软件与维护软件过程中遇到的一系列问题。

# 原因

  • 忽视开发前期需求分析。
  • 缺乏统一规范的指导。
  • 文档资料不全或不准。
  • 用户开发者缺乏交流。
  • 忽略测试重要性。
  • 不重视维护导致工作困难。
  • 专业人员缺乏经验。
  • 没有完善的质量保证体系。

# 软件工程三要素(课后题)

  • 过程
  • 方法
  • 工具

# 软件工程目标和原则(课后题)

# 目标

  • 达到要求的软件功能
  • 取得较好的软件性能
  • 开发出高质量的软件
  • 付出较低的开发成本
  • 需要较低的维护费用
  • 能按时完成工作，及时交付使用

# 原则

  1. 用分阶段的生命周期计划进行严格的管理
  2. 坚持进行阶段评审
  3. 实行严格的产品控制
  4. 采用现代化程序设计技术
  5. 软件工程结果应能被清楚的审查
  6. 开发小组人员应该少而精
  7. 承认不断改进工程实践的重要性

# 瀑布模型

  • 一种线性开发模型，具有不可回溯性。

# 优缺点

  • 优点：过程简单，执行容易。
  • 缺点：无法适应变更。

# 特点

  • 需求不发生或者很少变化，开发者一次性获得所有需求
  • 开发人员具有丰富的经验
  • 项目风险较低，不适合控制风险

# 快速原型模型

  • 优点：软件产品开发基本按线性顺序进行。
  • 缺点：所选用的开发技术和工具不一定符合主流的发展。

# 增量模型

  • 优点：将待开发软件系统模块化，一个开发周期内的错误不会影响到整个系统，开发顺序灵活。
  • 缺点：要求待开发软件系统可以被模块化。

# 螺旋模型

  • 优点：有助于把软件软件质量作为软件开发的一个重要目标。减少了过多测试或测试不足的风险。
  • 缺点：开发人员必须要有专业知识，否则会带来真正的风险。

# 喷泉模型

  • 特点：迭代和无缝。

  • 对程序有意义合理的输入数据的集合。

# 白盒测试

  • 被测程序看作是透明的盒子，关注产品内部细节和逻辑结构

# 代码检查法

  • 检查程序中代码设计的问题。

# 静态结构分析法

  • 一种对代码机械性的、程式化的特性进行分析的方法。

# 程序插桩技术

  • 借助程序中插入操作实现测试目的。

# 逻辑覆盖法(重点)

# 语句覆盖

  • 代码中所有的语句至少都执行一遍，用来检查测试用例是否有遗漏。

# 判定覆盖

  • 对模块中每一个判断各取真假各进行一次测试。

# 条件覆盖

  • 对每一个条件所有可能的取值都要执行一次。

# 条件判定覆盖

  • 同时满足条件覆盖与判定覆盖的逻辑覆盖。

# 条件组合覆盖

  • 判定条件中每一个条件的可能组合至少出现一次。

# 路径覆盖

  • 测试用例中执行到的路径数量占被测试模块所有可能的执行路径的比率。

# 白盒测试与黑盒测试的比较(重点)

黑盒测试 白盒测试
不涉及程序结构 考察程序逻辑结构
用软件说明书生成测试用例 用程序结构信息生成测试用例
适用于单元测试到系统验收测试 适用于单元测试和集成测试
某些代码段得不到测试 所有的逻辑都得到测试
  • 宏观黑盒测试，微观白盒测试。

# 软件测试的一般步骤

  • 着重测试每个单独模块
  • 必须把模块装配在一起形成完整的软件包
  • 软件集成完成后，还需进行高级测试
  • 高级测试最后一个步骤是系统测试

# 软件测试的阶段(课后题)

  • 单元、集成、确认、系统、验收

# 静态测试与动态测试区别(课后题)

# 静态测试

  • 不运行被测程序本身，通过分析检查语法、结构等检查程序正确性。

# 动态测试

  • 通过运行被测程序来检查结果与预计结果的差异。

# 软件维护(课后题)

  • 软件交付用户之后，为了弥补测试未发现的缺陷，改进性能补充新功能进行的修改软件过程

# 软件可维护性

  • 衡量对产品维护难易度的标准。

# 影响可维护性的因素

  • 理解性、可测试性、可修改性。

# 面向对象方法学与UML

# 面向对象方法的四个要点

  1. 认为客观世界是由各个对象组成的。
  2. 把所有对象都划分成各种对象类。
  3. 按照子类与父类的关系把若干个对象类组成一个有层次结构的系统。
  4. 对象彼此之间仅能通过传递消息，相互通信。

# 面向对象(课后题)

  • 对象是封装了数据结构及数据结构操作的封装体。

# 基本特点(课后题)

  • 以数据为中心
  • 主动性
  • 实现数据封装
  • 本质上具有并行性
  • 模块独立性好

# 类(课后题)

  • 类是具有相同数据相同操作的一组相似对象的定义。

# 面向对象优点(课后题)

  • 符合人类思维习惯
  • 稳定性好
  • 可复用性好
  • 可维护性好

# 对象模型

  • 类图是静态模型

# 可见性

  • 公有
  • 私有
  • 保护

# UML

  • UML——统一建模语言

# UML的设计原则

  • 模块化
  • 分层化
  • 分区化
  • 可扩展
  • 可重用

# UML优点(课后题)

  • 简洁直观，规范易理解，需求与设计分离，边界清晰。

# UML作用(课后题)

  • 为交流面向对象中的需求、行为、体系结构而出现的一套综合表示法。

# UML的图(课后题)

  • 用例图：提供功能描述。
  • 静态图：描述静态结构。
  • 行为图：描述动态行为和交互关系。
  • 交互图：描述对象间的交互关系。
  • 实现图：描述关于实现的信息。
